# Backend Architecture Specification - Wavo CRM

Wavo CRM's backend is a highly scalable REST API application built on **NestJS** and **Node.js**. It features a strict modular design, separation of concerns using the controller-service-repository pattern, type-safe database queries through **Prisma**, and asynchronous queue processing via **Redis** and **BullMQ**.

---

## 1. Directory Structure

The backend follows a domain-driven modular structure:

```
backend/
├── prisma/
│   ├── schema.prisma           # Complete relational database model definition
│   └── migrations/             # SQL schemas auto-generated by Prisma Migrations
├── src/
│   ├── main.ts                 # Application entry point & configuration
│   ├── app.module.ts           # Root module connecting all modules
│   ├── common/                 # Global utility blocks
│   │   ├── decorators/         # Custom decorators (e.g. @Roles(), @CurrentUser())
│   │   ├── guards/             # JWT authentication & RBAC guards
│   │   ├── filters/            # Global exception filters (maps database errors)
│   │   ├── interceptors/       # Logging, tenant isolation, serialization
│   │   └── dto/                # Global request schemas
│   ├── integrations/           # External service SDK initializers
│   │   ├── whatsapp.service.ts # Official Meta WhatsApp API integration
│   │   ├── stripe.service.ts   # Stripe billing and webhook event handling
│   │   └── gemini.service.ts   # Vertex AI / LLM api bindings
│   └── modules/                # Feature-specific modules
│       ├── auth/               # Session, registration, password hashing
│       ├── business/           # Multi-tenant details, GST setting, profile
│       ├── customers/          # Customer CRUD, timeline, tags
│       ├── leads/              # Sales stages, pipeline updates, Kanban logic
│       ├── followups/          # Reminder triggers, schedules, crons
│       ├── tasks/              # Task allocations, assignments, comments
│       └── campaigns/          # BullMQ queue creators, broadcast schedulers
└── package.json                # Project script execution and package management
```

## 2. Core Architectural Patterns

### 2.1 Controller - Service - Repository Pattern
To maintain testability and clean architecture, each feature module decouples routes, business logic, and database operations.

```
       +---------------------------------------------+
       |                  HTTP Request               |
       +----------------------+----------------------+
                              |
                              v
       +---------------------------------------------+
       |                  Controller                 |
       |  - Defines API endpoints (@Get, @Post)      |
       |  - Parameter validation via DTOs (class-val)|
       +----------------------+----------------------+
                              |
                              v
       +---------------------------------------------+
       |                    Service                  |
       |  - Coordinates business workflows           |
       |  - Invokes third-party APIs (Stripe, WA)    |
       +----------------------+----------------------+
                              |
                              v
       +---------------------------------------------+
       |                  Repository                 |
       |  - Decoupled database operations            |
       |  - Interacts directly with Prisma Client     |
       +----------------------+----------------------+
                              |
                              v
                      +---------------+
                      |  PostgreSQL   |
                      +---------------+
```

### 2.2 Strict Multi-Tenancy & Query Isolation
Data leakage is a critical threat in B2B SaaS. We prevent data leakage via:
*   **Prisma Client Middleware:** We configure a custom Prisma middleware that intercepts query operations and automatically appends a `where: { businessId }` clause to any query targetting customer, lead, invoice, or template data when executed under a standard user token.
*   **Tenant Interceptor:** Pulls the `businessId` from the verified JWT token payload and attaches it to the request scope (`request.businessId`), ensuring repositories consume it.

## 3. Security: Authentication & RBAC

### 3.1 Authentication
Authentication is stateless and uses short-lived JWT Access Tokens (signed with HS256) and long-lived Refresh Tokens stored in PostgreSQL.

### 3.2 Role-Based Access Control (RBAC)
We enforce permissions using custom guards matching role and permission tables:

```typescript
// roles.decorator.ts
export const Roles = (...roles: UserRoleName[]) => SetMetadata('roles', roles);

// roles.guard.ts
@Injectable()
export class RolesGuard implements CanActivate {
  constructor(private reflector: Reflector) {}

  canActivate(context: ExecutionContext): boolean {
    const requiredRoles = this.reflector.getAllAndOverride<UserRoleName[]>('roles', [
      context.getHandler(),
      context.getClass(),
    ]);
    if (!requiredRoles) return true;
    
    const { user } = context.switchToHttp().getRequest();
    return requiredRoles.includes(user.role);
  }
}
```
Usage in Controllers:
```typescript
@Patch(':id/stage')
@Roles(UserRoleName.BUSINESS_OWNER, UserRoleName.MANAGER, UserRoleName.SALES_EXECUTIVE)
async updateStage(@Param('id') id: string, @Body() dto: UpdateLeadStageDto) {
  return this.leadsService.updateStage(id, dto);
}
```

## 4. Background Processing (BullMQ & Redis)
High-throughput tasks such as **WhatsApp Broadcast campaigns** or **AI customer summaries** are processed out-of-band using Redis:
*   **Producer:** When a campaign is launched, the `CampaignsService` publishes a job containing recipient lists and templates to a Redis-backed queue (`whatsapp-broadcast`).
*   **Consumer:** A dedicated `BroadcastWorker` process picks up jobs from the queue, handles rate-limiting, interacts with Meta's endpoint, updates execution logs, and supports retry intervals on failures.
